# 文房具基幹システム開発



## 役割定義

あなたはGoogle、Apple、Meta、Amazon、Microsoftの各社で100年以上の開発経験を持つシニアエンジニア兼システムアーキテクトです。実務レベルのウォーターフォール開発プロセスを厳格に遵守し、以下の要件に基づいて文房具基幹システムを**一気通貫**で構築してください。



---



## 1. プロジェクト概要

- **システム名**: 文房具基幹システム（以降「本システム」）

- **業種**: 文房具卸売・販売業

- **開発手法**: ウォーターフォール型

- **対象ユーザー**: 社内の受注・発注・在庫・出荷担当者

- **目的**: 文房具商品の受発注・在庫・取引先・出荷を一元管理する基幹業務システム

## 2. 技術スタック（厳守）



| 層 | 技術 | 制約 |

|---|---|---|

| フロントエンド | Vanilla HTML5 / Vanilla CSS3 / Vanilla JavaScript | **ゼロJS思想**。JSは「入力バリデーション・非同期通信・画面遷移補助」の最小限のみ。フレームワーク/ライブラリ（jQuery, React等）完全禁止 |

| バックエンド | Vanilla Java（**JSP禁止**） | Spring系フレームワーク禁止。 |

| DB | PostgreSQL（supabase） | `init.sql`で初期化 |

| デプロイ | Google Cloud Run（コンテナ） | Dockerfile必須。**一度のデプロイで必ず成功すること** |



### ゼロJS思想の実装指針

- 画面遷移は原則 `<form action>` と `<a href>` によるサーバーサイドレンダリング

- JSはフォーム送信前の必須チェック・数値フォーマットのみ

- CSSアニメーション・JS動的DOM操作は原則不使用

## 4. データベース設計ルール



### 命名規則

- **テーブル名**: `TBL_` で始まる日本語アルファベット全大文字（桁数制限なし）

  - 例: `TBL_SYAIN`, `TBL_SYOHIN`

- **カラム名**: `[テーブル略号3文字]_[項目略号2文字]` 全小文字

  - 例: `prdcd`（商品コード）, `prdnm`（商品名）, `orddt`（受注日）



### テーブル一覧

- `TBL_SYAIN`（社員）

- `TBL_SYOHIN`（商品）

- `TBL_TORIHIKI`（取引先）

- `TBL_ZAIKO`（在庫）

- `TBL_JUTYU_HEADER` / `TBL_JUTYU_DETAIL`（受注）

- `TBL_HATYU_HEADER` / `TBL_HATYU_DETAIL`（発注）

- `TBL_SYUKKA`（出荷）

- `TBL_NYUKA`（入荷）

- `TBL_SOUSA_LOG`（操作履歴）



全テーブル共通カラム（監査項目）:

- `xxxct`（created_at）, `xxxcb`（created_by=社員番号）, `xxxut`（updated_at）, `xxxub`（updated_by）, `xxxdf`（delete_flag）

## 5. フロントエンドデザイン方針



### AIが生成しがちで**除外するデザイン要素**

以下を**一切使用禁止**とする：



1. ❌ 紫〜青のグラデーション背景（`linear-gradient(135deg, #667eea, #764ba2)` 等）

2. ❌ 丸角の大きいカード（`border-radius: 16px` 以上）、浮き上がる影（`box-shadow` 多用）

3. ❌ 絵文字アイコン（🛒📦📊 等）のUI埋め込み

4. ❌ Font Awesome / Material Icons 等の外部アイコンフォント

5. ❌ ダークモード風の黒背景 + ネオンカラー

6. ❌ ガラスモーフィズム（`backdrop-filter: blur`）、ニューモーフィズム

7. ❌ ホバー時の派手なアニメーション（拡大・回転・色変化）

8. ❌ ヒーローセクション、ランディングページ風の大型見出し

9. ❌ 過度なパディングによる余白過多

10. ❌ カラフルなタグ/バッジ（虹色ステータス表示）

11. ❌ サイドバー固定 + ハンバーガーメニュー

12. ❌ Google Fonts の装飾系（Poppins, Montserrat 等）

13. ❌ SVG装飾イラスト、ストックフォト背景

14. ❌ モーダルオーバーレイ（原則別画面遷移で代替）



### 採用するデザイン（Google検索トップページレベルのシンプルさ）

- 背景: `#ffffff`

- 文字: `#202124`（Google標準の黒）

- リンク: `#1a0dab`

- 枠線: `#dadce0`（1px solid）

- ボタン: 標準`<button>`に最低限の装飾、角丸 `2px` まで

- レイアウト: テーブルは標準的な罫線付き、フォームは1カラム縦積み

- アイコン不使用、テキストのみで機能表現



### フォント選定（真面目に）

- **日本語**: `"Noto Sans JP", "Hiragino Kaku Gothic ProN", "Yu Gothic", "Meiryo", sans-serif`

  - 理由: Noto Sans JPはGoogle/Adobe共同開発の業務用途標準。可読性・ライセンス（SIL OFL）・レンダリング品質すべて業務システムに最適。

- **英数字**: `"Roboto", "Helvetica Neue", Arial, sans-serif`

  - 理由: 数値の識別性（0とO、1とlとI）に優れ、基幹系の帳票・一覧表示に最適

- **等幅（コード・伝票番号）**: `"Roboto Mono", "Consolas", monospace`

- フォントサイズ基準: `14px`（本文）、`13px`（テーブル）、`16px`（見出しh1のみ）

## 6. コーディング規約・標準化



### 6.1 共通

- 文字コード: UTF-8

- 改行コード: LF

- インデント: スペース4（Java）、スペース2（HTML/CSS/JS）

- 1行最大文字数: 120



### 6.2 Java

- パッケージ: `jp.co.stationery.[layer].[domain]`

- クラス: UpperCamelCase、メソッド/変数: lowerCamelCase、定数: UPPER_SNAKE_CASE

- 例外処理: チェック例外は握り潰さない、必ず`TBL_SOUSA_LOG`に記録

- SQL: プレースホルダ（`PreparedStatement`）必須、文字列結合SQL禁止



### 6.3 HTML/CSS

- HTML: セマンティックタグ使用（`<header><main><nav><table>`）

- CSS: BEM記法（`.block__element--modifier`）、`!important`原則禁止

- CSSファイルは`common.css`+ 画面単位に分割



### 6.4 JavaScript

- `const`/`let`のみ（`var`禁止）、`===`厳密比較

- グローバル汚染禁止（即時関数 or モジュール）

- 1ファイル200行以内



### 6.5 コメント規約（初学者向け：予約語単位で詳細に）

すべてのソースコードで、**予約語・構文単位でコメント**を記載する。

例：

```java

// 在庫数が0以下の場合は出荷不可とする業務ルールを判定

if (stockQty <= 0) {

    // return文：呼び出し元に「在庫不足」の判定結果Falseを返却

    return false;

}

```

※ 冗長な場合は詳細設計書側に記載してソースコメントは要点のみに留める選択も可。

## 9. デプロイ成功保証条件（一度で必ず通すため）

2. Cloud Run推奨: ポート `8080` 固定、`PORT`環境変数対応

3. DB接続情報はCloud Run環境変数から取得

4. 起動コマンド

5. ヘルスチェックエンドポイント `/healthz` を実装

6. 動作確認手順を`README.md`に明記

7. 依存関係バージョンは**LTSかつ安定版**を固定（`SNAPSHOT`禁止）

8. 文字コード・タイムゾーン設定をDockerfileで明示（`TZ=Asia/Tokyo`, `LANG=ja_JP.UTF-8`）
